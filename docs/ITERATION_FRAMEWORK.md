# 반복 개선 프레임워크

## 목적

하네스 엔지니어링은 한 번 설정하고 끝나는 것이 아니라 지속적으로 개선되어야 한다.
이 문서는 문서 하네스를 반복적으로 검증하고 개선하는 프레임워크를 정의한다.

## 핵심 원칙

### 1. 측정 가능한 개선

**나쁜 목표:**
- "문서를 더 잘 쓰자"
- "코드 품질을 높이자"

**좋은 목표:**
- "설계 문서 없는 PR을 0건으로 만들자"
- "계획 문서의 검증 기준 충족률을 100%로 만들자"

### 2. 짧은 피드백 루프

**주간 → 월간 → 분기별**
- 빠른 피드백으로 문제 조기 발견
- 누적된 문제가 커지기 전에 해결

### 3. 작은 변경, 잦은 실험

- 프레임워크 자체도 점진적으로 개선
- 실험 → 측정 → 회고 → 적용

## 반복 주기

### 일일 점검 (Daily Check)

**대상:** 작업 중인 개인 또는 Agent

**체크리스트:**
- [ ] 오늘 작업과 연결된 문서를 확인했는가?
- [ ] 계획 문서가 있다면 진행 상황을 갱신했는가?
- [ ] 예상과 다른 점이 있다면 문서에 기록했는가?
- [ ] 새로운 리스크나 가정이 생겼다면 즉시 기록했는가?

**소요 시간:** 5분 이내

**목적:** 문서와 코드의 동기화 상태 유지

### 주간 리뷰 (Weekly Review)

**대상:** 팀 또는 주요 기여자

**리뷰 항목:**

1. **활성 계획 상태**
   - `docs/exec-plans/active/`의 모든 계획 검토
   - 진행 중인 계획이 실제 작업과 일치하는가?
   - 방치된 계획이 있는가?
   - 완료된 계획을 `completed/`로 이동했는가?

2. **문서-코드 정합성**
   - 이번 주 변경된 코드에 대응하는 문서가 있는가?
   - 설계 문서가 실제 구현과 일치하는가?
   - 제품 스펙이 배포된 기능과 일치하는가?

3. **부채 누적**
   - `docs/exec-plans/tech-debt-tracker.md` 확인
   - 새로운 부채 항목이 추가되었는가?
   - 기존 부채 중 해결된 것이 있는가?
   - 우선순위 재평가가 필요한가?

4. **품질 게이트 통과율**
   - 이번 주 완료된 작업의 품질 점수
   - 검증 기준 충족률
   - 보안/신뢰성 체크리스트 통과율

**체크리스트:**
```markdown
## 주간 리뷰 YYYY-MM-DD

### 활성 계획
- [ ] 모든 활성 계획이 진행 중인가?
- [ ] 완료된 계획을 completed/로 이동했는가?
- [ ] 새 계획이 필요한 작업이 있는가?

### 문서-코드 정합성
- [ ] 설계 문서와 코드가 일치하는가?
- [ ] 제품 스펙과 기능이 일치하는가?
- [ ] 생성 문서 (스키마, API)가 최신인가?

### 부채 관리
- [ ] 새로운 부채가 트래커에 기록되었는가?
- [ ] 해결된 부채를 제거했는가?
- [ ] 고위험 부채에 대한 계획이 있는가?

### 품질 메트릭
- 완료된 작업 수: X건
- 품질 점수 평균: X.X / 5.0
- 검증 기준 충족률: XX%
- 문서 없는 변경: X건 (목표: 0건)
```

**소요 시간:** 30분 ~ 1시간

**산출물:**
- 주간 리뷰 노트 (`docs/reviews/weekly/YYYY-MM-DD.md`)
- 개선 액션 아이템

### 월간 감사 (Monthly Audit)

**대상:** 전체 팀 또는 프로젝트 리더

**감사 영역:**

#### 1. 문서 무결성

**링크 검증:**
```bash
# markdown-link-check로 깨진 링크 찾기
npx markdown-link-check docs/**/*.md
```

**중복 검출:**
- 유사한 내용을 다루는 문서가 여러 개 있는가?
- 통합 또는 정리 기회가 있는가?

**고아 문서:**
- 어디서도 링크되지 않은 문서가 있는가?
- 인덱스 문서에서 누락된 항목이 있는가?

#### 2. 설계 문서 품질

**각 설계 문서 검토:**
- [ ] 문제 정의가 명확한가?
- [ ] 결정 내용이 구체적인가?
- [ ] 고려한 대안이 기록되어 있는가?
- [ ] 리스크가 명시되어 있는가?
- [ ] 코드와 일치하는가?

**개선 필요 항목:**
- 내용이 부실한 문서 목록
- 세분화가 필요한 문서
- 통합이 필요한 문서

#### 3. 제품 스펙 품질

**각 제품 스펙 검토:**
- [ ] 사용자 관점이 명확한가?
- [ ] 입력/출력이 정의되어 있는가?
- [ ] 상태 전이가 명확한가?
- [ ] 비즈니스 규칙이 구체적인가?
- [ ] 배포된 기능과 일치하는가?

#### 4. 생성 문서 동기화

**자동 생성 문서 재생성:**
```bash
# 스키마 재생성 예시
python scripts/generate_schema_doc.py > docs/generated/db-schema.md

# API 문서 재생성 예시
python scripts/generate_openapi.py > docs/generated/api-openapi.yaml
```

**확인:**
- [ ] 생성 문서가 코드와 일치하는가?
- [ ] 변경된 부분이 명확히 표시되는가?

#### 5. 부채 트래커 정리

**우선순위 재평가:**
- 각 부채 항목의 영향도 재평가
- 비즈니스 우선순위 변경 반영
- 해결된 항목 제거
- 새로운 항목 추가

**분류:**
- 긴급 (1주 이내 해결)
- 높음 (1개월 이내 해결)
- 중간 (3개월 이내 해결)
- 낮음 (기회가 있을 때 해결)

#### 6. 참고 자료 최신성

**외부 링크 검증:**
- 참고 문서의 외부 링크가 유효한가?
- 참고한 라이브러리 버전이 최신인가?
- 업데이트가 필요한 참고 자료가 있는가?

**체크리스트:**
```markdown
## 월간 감사 YYYY-MM

### 문서 무결성
- [ ] 깨진 링크: X건 (수정 완료)
- [ ] 중복 문서: X건 (통합 계획)
- [ ] 고아 문서: X건 (링크 추가 또는 삭제)

### 설계 문서
- [ ] 전체 문서 수: X건
- [ ] 개선 필요: X건
- [ ] 코드 불일치: X건 (갱신 완료)

### 제품 스펙
- [ ] 전체 스펙 수: X건
- [ ] 개선 필요: X건
- [ ] 기능 불일치: X건 (갱신 완료)

### 생성 문서
- [ ] db-schema.md 갱신 완료
- [ ] api-openapi.yaml 갱신 완료
- [ ] 기타 생성 문서 갱신 완료

### 부채 트래커
- [ ] 전체 부채: X건
- [ ] 긴급: X건
- [ ] 높음: X건
- [ ] 해결됨: X건

### 참고 자료
- [ ] 외부 링크 유효성 확인
- [ ] 라이브러리 버전 갱신 필요: X건
```

**소요 시간:** 2~3시간

**산출물:**
- 월간 감사 리포트 (`docs/reviews/monthly/YYYY-MM.md`)
- 개선 계획 문서
- 정리된 부채 트래커

### 분기별 회고 (Quarterly Retrospective)

**대상:** 전체 팀

**회고 질문:**

#### 1. 효과성 평가

**문서 하네스가 도움이 되었는가?**
- 의사결정 속도가 빨라졌는가?
- 컨텍스트 전환 비용이 줄어들었는가?
- 신규 팀원 온보딩이 쉬워졌는가?
- AI Agent 활용이 효율적이었는가?

**정량적 지표:**
- 문서 있는 PR 비율: XX%
- 계획 문서 작성률: XX%
- 검증 기준 충족률: XX%
- 평균 품질 점수: X.X / 5.0

#### 2. 문서 유형별 유용성

**가장 유용했던 문서:**
- 설계 문서: 어떤 종류가 가장 많이 참조되었는가?
- 제품 스펙: 어떤 스펙이 가장 명확했는가?
- 실행 계획: 어떤 계획이 가장 잘 작동했는가?

**개선이 필요한 문서:**
- 작성했지만 거의 참조되지 않은 문서
- 형식적으로만 작성된 문서
- 유지보수가 부담스러운 문서

#### 3. 프로세스 효율성

**병목 지점:**
- 문서 작성에 시간이 너무 오래 걸리는가?
- 문서 갱신이 자주 누락되는가?
- 문서 찾기가 어려운가?

**개선 기회:**
- 자동화 가능한 부분이 있는가?
- 템플릿을 개선할 수 있는가?
- 도구 지원이 필요한가?

#### 4. 구조 개선

**디렉토리 구조:**
- 현재 구조가 직관적인가?
- 새로운 문서 유형이 필요한가?
- 통합하거나 분리할 디렉토리가 있는가?

**문서 연결:**
- 링크 구조가 명확한가?
- 인덱스 문서가 효과적인가?
- 네비게이션 개선 방법이 있는가?

#### 5. 규칙 유효성

**지켜진 규칙:**
- 어떤 규칙이 자연스럽게 지켜졌는가?
- 어떤 규칙이 도움이 되었는가?

**무시된 규칙:**
- 어떤 규칙이 자주 무시되었는가?
- 왜 무시되었는가? (너무 엄격? 불명확? 불필요?)
- 완화하거나 제거해야 하는가?

**회고 워크시트:**
```markdown
## 분기별 회고 YYYY-QX

### 효과성 평가
- 의사결정 속도: [1-5점]
- 컨텍스트 전환 비용: [1-5점]
- 온보딩 용이성: [1-5점]
- AI Agent 효율성: [1-5점]

정량 지표:
- 문서 있는 PR: XX%
- 계획 문서 작성률: XX%
- 검증 기준 충족률: XX%
- 평균 품질 점수: X.X / 5.0

### 문서 유형별 평가
가장 유용:
- 설계 문서: [구체적 예시]
- 제품 스펙: [구체적 예시]
- 실행 계획: [구체적 예시]

개선 필요:
- [문서 이름]: [이유]
- [문서 이름]: [이유]

### 프로세스 병목
- [병목 1]: [해결 방안]
- [병목 2]: [해결 방안]

### 구조 개선 제안
- 디렉토리: [제안]
- 링크 구조: [제안]
- 네비게이션: [제안]

### 규칙 검토
지켜진 규칙:
- [규칙 1]: 효과적
- [규칙 2]: 효과적

무시된 규칙:
- [규칙 1]: [이유] → [조치: 완화/제거/명확화]
- [규칙 2]: [이유] → [조치: 완화/제거/명확화]

### 다음 분기 목표
1. [목표 1] - 측정 지표: [X]
2. [목표 2] - 측정 지표: [X]
3. [목표 3] - 측정 지표: [X]
```

**소요 시간:** 반나절 (팀 세션)

**산출물:**
- 분기별 회고 리포트 (`docs/reviews/quarterly/YYYY-QX.md`)
- 하네스 개선 계획 문서
- 다음 분기 목표와 지표

## 측정 지표

### 선행 지표 (Leading Indicators)

작업 진행 중 측정 가능한 지표:

| 지표 | 목표 | 측정 방법 |
|-----|-----|---------|
| 문서 있는 PR 비율 | 100% | PR 메타데이터에 문서 링크 존재 |
| 계획 문서 작성률 | 100% (3단계 이상 작업) | exec-plans/active/ 파일 존재 |
| 검증 기준 명시율 | 100% | 계획 문서에 "검증 전략" 섹션 존재 |
| 일일 문서 갱신 | 작업일 80% 이상 | Git 커밋 로그 분석 |

### 후행 지표 (Lagging Indicators)

작업 완료 후 측정 가능한 지표:

| 지표 | 목표 | 측정 방법 |
|-----|-----|---------|
| 평균 품질 점수 | 4.0 / 5.0 이상 | QUALITY_SCORE.md 평가표 적용 |
| 문서-코드 일치율 | 95% 이상 | 월간 감사 시 수동 검증 |
| 부채 해결률 | 분기당 30% 이상 | tech-debt-tracker.md 추적 |
| 문서 참조 빈도 | 주당 팀원당 10회 이상 | Git 로그 또는 설문 |

### 건강도 지표 (Health Metrics)

시스템 전체 건강도:

| 지표 | 목표 | 의미 |
|-----|-----|-----|
| 깨진 링크 | 0건 | 문서 무결성 |
| 고아 문서 | 0건 | 문서 연결성 |
| 방치된 활성 계획 | 0건 (7일 이상 갱신 없음) | 계획 실행력 |
| 생성 문서 지연 | 0일 (코드 변경 후 즉시 갱신) | 동기화 상태 |

## 개선 실험 프로세스

### 가설 기반 개선

**형식:**
```markdown
## 개선 실험 YYYY-MM-DD

### 가설
[현재 문제] 때문에 [부정적 영향]이 발생한다.
[개선 방안]을 적용하면 [긍정적 결과]를 얻을 것이다.

### 측정 지표
- 현재 상태: [X]
- 목표 상태: [Y]
- 측정 방법: [Z]

### 실험 기간
YYYY-MM-DD ~ YYYY-MM-DD (N주)

### 실험 내용
[구체적 변경 사항]

### 롤백 계획
[실패 시 원복 방법]
```

**예시:**
```markdown
## 개선 실험 2026-04-01

### 가설
계획 문서 작성이 부담스러워서 작성률이 낮다(현재 60%).
계획 문서 템플릿을 간소화하면 작성률이 90% 이상으로 올라갈 것이다.

### 측정 지표
- 현재 계획 문서 작성률: 60%
- 목표 작성률: 90%
- 측정: 4주간 exec-plans/active/ 파일 생성 비율

### 실험 기간
2026-04-01 ~ 2026-04-28 (4주)

### 실험 내용
1. PLANS.md 템플릿에서 "롤아웃 메모" 섹션 제거
2. "작업 분해"와 "검증 전략"을 통합
3. 최소 길이 제한 제거

### 롤백 계획
4주 후 작성률이 70% 미만이면 원래 템플릿으로 복구
```

### 실험 결과 기록

**형식:**
```markdown
## 실험 결과 YYYY-MM-DD

### 가설 재진술
[원래 가설]

### 측정 결과
- 목표: [Y]
- 실제: [Z]
- 차이: [Z - Y]

### 관찰
- [관찰 1]
```

### 생성 문서 자동 갱신

**Git Hook 예시: `.git/hooks/pre-commit`**
```bash
#!/bin/bash

# 모델 파일 변경 시 스키마 문서 재생성
if git diff --cached --name-only | grep -q "app/models/"; then
  echo "모델 변경 감지 - 스키마 문서 재생성 중..."
  python scripts/generate_schema_doc.py > docs/generated/db-schema.md
  git add docs/generated/db-schema.md
fi

# API 라우터 변경 시 OpenAPI 스펙 재생성
if git diff --cached --name-only | grep -q "app/api/"; then
  echo "API 변경 감지 - OpenAPI 스펙 재생성 중..."
  python scripts/generate_openapi.py > docs/generated/api-openapi.yaml
  git add docs/generated/api-openapi.yaml
fi
```

### 주간 리뷰 리마인더

**GitHub Actions 예시: `.github/workflows/weekly-review.yml`**
```yaml
name: Weekly Review Reminder

on:
  schedule:
    - cron: '0 9 * * 1'  # 매주 월요일 오전 9시

jobs:
  remind:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Check for weekly review
        run: |
          LAST_REVIEW=$(ls -t docs/reviews/weekly/*.md | head -1)
          LAST_DATE=$(basename $LAST_REVIEW .md)
          DAYS_AGO=$(( ($(date +%s) - $(date -d $LAST_DATE +%s)) / 86400 ))

          if [ $DAYS_AGO -gt 7 ]; then
            echo "⚠️ 주간 리뷰가 $DAYS_AGO일 지연되었습니다!"
            # Slack/Discord 알림 전송
          fi
```
