# 하네스 엔지니어링 가이드

## 목적

하네스 엔지니어링은 코드보다 문서와 구조를 먼저 설계하여, AI Agent든 인간이든 일관되고 예측 가능한 방식으로 시스템을 발전시키는 방법론이다.

이 문서는 어떤 AI 도구를 사용하든 관계없이 적용 가능한 원칙과 프로세스를 정의한다.

## 핵심 원칙

### 1. 문서 우선주의 (Documentation-First)

```
변경 → 문서 → 코드 (X)
문서 → 변경 계획 → 코드 → 검증 → 문서 갱신 (O)
```

**규칙:**
- 코드 작성 전에 관련 문서 위치를 확인한다
- 구조적 결정은 설계 문서로 먼저 명문화한다
- 제품 동작 변경은 제품 스펙으로 먼저 정의한다
- 실행은 계획 문서로 작업을 분해하고 검증 기준을 명시한다

### 2. 경계 명확성 (Clear Boundaries)

**각 문서 유형의 책임:**
- 제품 스펙 → 사용자 관점 요구사항과 행동
- 도메인 문서 → 도메인 개념과 테스트 보장 범위
- 설계 문서 → 기술 결정과 아키텍처 근거
- 실행 계획 → 작업 분해, 순서, 검증 전략
- 생성 문서 → 코드/모델에서 파생된 사실
- 학습 문서 → 공식 정본이 아닌 학습/실험 메모

**금지:**
- 제품 스펙에 구현 세부사항 작성
- 제품 스펙에 API 필드, 상태 enum, 테스트 보장 범위 작성
- 설계 문서에 일회성 진행 상황 기록
- 실행 계획에 장기 아키텍처 결정 혼입
- 학습 문서를 정본 문서처럼 참조

### 3. 검증 가능성 (Verifiability)

**모든 작업은 검증 기준과 함께:**
- "구현했다" → "어떻게 확인했는가?"
- "동작한다" → "어떤 조건에서 테스트했는가?"
- "완료했다" → "문서와 실제가 일치하는가?"

**최소 검증 수준:**
- 단위 기능: 로컬 테스트 또는 수동 확인
- API 변경: 계약 테스트 또는 예제 요청
- 도메인 로직: 경계 조건과 에러 케이스 포함
- 보안/신뢰성: 체크리스트 기반 검토

### 4. 점진적 진화 (Incremental Evolution)

**작은 변경, 잦은 검증:**
```
대규모 리팩토링 (X)
→ 기존 패턴 재사용 → 작은 변경 → 검증 → 다음 단계 (O)
```

**변경 크기 원칙:**
- 되돌릴 수 있는 단위로 끊는다
- 파급 영향을 문서로 예측한다
- 새 추상화보다 기존 구조 재사용을 우선한다

### 5. 근거 기반 결정 (Evidence-Based Decision)

**추정보다 사실:**
- "아마도 이렇게 동작할 것" (X)
- "이 문서/코드/테스트로 확인했다" (O)

**근거 수집 순서:**
1. 기존 문서 확인
2. 코드 읽기 또는 실험
3. 외부 참고 자료 검색
4. 근거가 부족하면 가정을 명시하고 나중에 재검토

## 문서 하네스 구조

### 계층적 진입점

```
README.md (프로젝트 개요)
├─ AGENTS.md (Agent 작업 규칙)
├─ ARCHITECTURE.md (시스템 지도)
└─ docs/
   ├─ DESIGN.md (설계 문서 작성 규칙)
   ├─ PLANS.md (계획 문서 운영 방식)
   ├─ PRODUCT_SENSE.md (제품 우선순위 판단)
   ├─ QUALITY_SCORE.md (품질 평가 기준)
   ├─ RELIABILITY.md (신뢰성 기준)
   ├─ SECURITY.md (보안 기준)
   └─ HARNESS_ENGINEERING.md (본 문서)
```

### 문서 디렉토리 분류

| 디렉토리 | 책임 | 수명 |
|---------|-----|-----|
| `docs/design-docs/` | 기술 결정, 아키텍처 근거 | 영구 |
| `docs/product-specs/` | 사용자 요구사항, 제품 동작 | 영구 |
| `docs/domains/` | 도메인 개념, 책임, 테스트 보장 범위 | 영구 |
| `docs/exec-plans/active/` | 진행 중 작업 계획 | 임시 |
| `docs/exec-plans/completed/` | 완료된 작업 기록 | 보관 |
| `docs/generated/` | 코드/모델/설계 정본에서 파생된 스키마 스냅샷 | 파생 |
| `docs/references/` | 외부 도구, 플랫폼 메모 | 참고 |
| `docs/study/` | 학습, 실험, 비정본 메모 | 참고 |

## 작업 흐름 프로세스

### Phase 1: 진입 (Entry)

**질문:**
1. 이 작업은 어떤 문서와 연결되는가?
2. 관련 설계나 스펙이 존재하는가?
3. 기존 패턴이나 구조를 재사용할 수 있는가?

**행동:**
- `ARCHITECTURE.md`에서 시스템 개요 확인
- `docs/design-docs/index.md`에서 관련 설계 문서 탐색
- `docs/product-specs/index.md`에서 제품 스펙 확인
- `docs/domains/README.md`에서 도메인 개념과 테스트 보장 확인
- `docs/exec-plans/tech-debt-tracker.md`에서 부채 항목 체크

### Phase 2: 계획 (Planning)

**문서 작성:**
1. 변경이 제품 동작에 영향을 주면 `docs/product-specs/`에 스펙 작성 또는 갱신
2. 구조적 결정이면 `docs/design-docs/`에 설계 문서 작성
3. 실행 순서와 검증 전략을 `docs/exec-plans/active/`에 계획 문서 작성

**계획 문서 필수 항목:**
```markdown
## 목표
- 명확하고 검증 가능한 목표

## 범위와 비범위
- 이 계획이 다루는 것
- 의도적으로 제외하는 것

## 가정
- 전제 조건
- 불확실한 항목

## 작업 분해
1. [작업 1]
2. [작업 2]
   - 검증 기준: [구체적 확인 방법]

## 검증 전략
- 단위 검증
- 통합 검증
- 수동 확인 체크리스트

## 롤아웃 메모
- 배포 순서
- 롤백 시나리오

## 남은 리스크
- 알려진 제약
- 후속 작업
```

### Phase 3: 실행 (Execution)

**규칙:**
- 계획 문서의 작업 순서를 따른다
- 각 작업 후 검증 기준을 확인한다
- 예상과 다르면 계획 문서를 갱신한다
- 새로운 리스크나 부채는 즉시 기록한다

**실행 중 문서 갱신:**
- 설계가 바뀌면 설계 문서 갱신
- 제품 동작이 바뀌면 제품 스펙 갱신
- 영속성 구조가 바뀌면 파생 스냅샷인 `docs/generated/db-schema.md`를 갱신
- 도메인별 테스트 보장 범위가 바뀌면 `docs/domains/*/test-scenarios.md`를 갱신

### Phase 4: 검증 (Verification)

**체크리스트:**
- [ ] 계획 문서의 모든 작업이 완료되었는가?
- [ ] 각 작업의 검증 기준을 충족했는가?
- [ ] 관련 문서가 실제 구현과 일치하는가?
- [ ] 품질 점수 기준을 통과하는가? (`docs/QUALITY_SCORE.md`)
- [ ] 보안 체크리스트를 확인했는가? (`docs/SECURITY.md`)
- [ ] 신뢰성 요구사항을 충족하는가? (`docs/RELIABILITY.md`)

### Phase 5: 완료 (Completion)

**문서 정리:**
1. 계획 문서를 `docs/exec-plans/active/`에서 `docs/exec-plans/completed/`로 이동
2. 남은 부채나 후속 작업을 `docs/exec-plans/tech-debt-tracker.md`에 기록
3. 상위 인덱스 문서 (`design-docs/index.md`, `product-specs/index.md`) 갱신
4. `ARCHITECTURE.md`가 여전히 정확한지 확인

**커밋 메시지:**
```
[Category] Brief description

- Related docs: docs/exec-plans/completed/YYYY-MM-DD-title.md
- Verification: [how it was verified]
- Follow-ups: [links to tech debt tracker or new plans]
```

## 반복 개선 프레임워크

### 주간 점검 (Weekly Review)

**확인 항목:**
1. 활성 계획 문서가 실제 작업과 동기화되어 있는가?
2. 완료된 계획이 `completed/`로 이동되었는가?
3. 부채 트래커에 새로운 항목이 누적되고 있는가?
4. 설계 문서와 코드가 여전히 일치하는가?

### 월간 감사 (Monthly Audit)

**점검 영역:**
- 문서 링크 무결성 검사
- 부채 트래커 우선순위 재평가
- 설계 문서 정리 (중복 제거, 통합)
- 생성 문서 재생성 (스키마, API 계약)

### 분기별 회고 (Quarterly Retrospective)

**질문:**
1. 문서 하네스가 의사결정 속도를 높였는가?
2. 어떤 문서 유형이 가장 유용했는가?
3. 어떤 문서가 방치되고 있는가?
4. 구조 개선 기회는 무엇인가?

**산출물:**
- 하네스 구조 개선 계획
- 문서 템플릿 갱신
- 가드레일 추가 또는 완화

## AI Agent 활용 시 가이드라인

### Agent에게 요청할 때

**좋은 요청:**
```
"docs/product-specs/new-feature.md 스펙에 따라 API를 구현하고,
docs/exec-plans/active/에 실행 계획을 먼저 작성해줘"
```

**나쁜 요청:**
```
"그냥 새 기능 만들어줘" (문서 연결 없음)
```

### Agent가 지켜야 할 규칙

1. 작업 시작 전 관련 문서 확인
2. 불확실하면 가정을 명시하고 사용자에게 확인
3. 계획 문서 작성 후 코드 구현
4. 검증 근거와 함께 완료 보고
5. 문서와 코드가 불일치하면 문서를 먼저 갱신

### Agent 독립성 원칙

**하네스는 Agent에 독립적:**
- Claude Code, Cursor, GitHub Copilot 등 어떤 도구든 동일한 문서 구조 사용
- Agent 간 전환 시 문서만 보면 컨텍스트 복원 가능
- Agent가 바뀌어도 문서 하네스는 유지됨

## 품질 게이트 (Quality Gates)

### 머지 전 필수 확인

- [ ] 관련 문서가 존재하고 최신 상태인가?
- [ ] 계획 문서의 검증 기준을 충족했는가?
- [ ] 품질 점수 평가표에서 모든 항목이 3점 이상인가?
- [ ] 보안 체크리스트를 통과했는가?
- [ ] 남은 리스크가 부채 트래커에 기록되었는가?

### 배포 전 필수 확인

- [ ] 생성 문서 (스키마, API 계약)가 최신인가?
- [ ] 롤백 계획이 문서화되어 있는가?
- [ ] 운영 매뉴얼이나 모니터링 지침이 있는가?
- [ ] 외부 의존성 변경이 참고 문서에 반영되었는가?

## 안티패턴 (Anti-Patterns)

### 1. 문서 없는 코드 (Orphan Code)
**문제:** 코드만 쌓이고 왜 그렇게 만들었는지 알 수 없음
**해결:** 설계 문서 또는 계획 문서와 연결 강제

### 2. 낡은 문서 (Stale Documentation)
**문제:** 문서와 코드가 불일치하여 혼란 가중
**해결:** 변경 시 문서 갱신을 완료 조건에 포함

### 3. 계획 없는 실행 (Cowboy Coding)
**문제:** 큰 변경을 일단 저지르고 나중에 정리
**해결:** 계획 문서 작성을 작업 시작 조건으로 설정

### 4. 검증 없는 완료 (Fake Completion)
**문제:** "아마 동작할 것"으로 완료 처리
**해결:** 검증 기준 명시와 실제 확인 강제

### 5. 무한 문서화 (Documentation Paralysis)
**문제:** 문서만 쓰고 코드는 안 씀
**해결:** 문서는 최소 필수 항목만, 실행은 빠르게

## 도구 추천

### 문서 작성
- Markdown 에디터
- 링크 검증 도구 (markdown-link-check)
- 다이어그램 (Mermaid, PlantUML)

### 검증 자동화
- 문서-코드 불일치 탐지 스크립트
- 스키마 diff 도구
- API 계약 검증 (OpenAPI validator)

### 버전 관리
- Git으로 문서와 코드 동시 관리
- 문서 변경도 PR 리뷰 대상
