# 설계 가이드

## 목적

`docs/design-docs/`는 구조, 경계, 유지보수 비용에 영향을 주는 기술 결정을 기록하는 공간이다.

## 언제 설계 문서를 쓰는가

아래 항목 중 하나 이상에 영향이 있으면 설계 문서를 새로 쓰거나 갱신한다.

- 도메인 경계
- API 형태와 호환성 규칙
- 영속성 모델과 마이그레이션 방향
- 신뢰성 또는 보안 구조
- 여러 모듈에 걸쳐 재사용될 구현 패턴

## 최소 구성

각 설계 문서는 아래 내용을 포함한다.

1. 문제 정의
2. 배경과 제약
3. 결정 내용
4. 고려한 대안
5. 롤아웃 또는 마이그레이션 메모
6. 리스크와 열린 질문

## 저장소 규칙

- 설계 문서는 `docs/design-docs/`에 둔다.
- 관련 제품 스펙이나 실행 계획이 있으면 함께 링크한다.
- 유사한 새 문서를 늘리기보다 기존 문서를 갱신하는 쪽을 우선한다.
- 설계 문서에는 일회성 진행 상황보다 오래 유지될 근거를 남긴다.

## DTO와 서비스 네이밍

- 외부 API 입력 모델은 `*Request`로 끝낸다. Body와 Query parameter를 묶는 요청 모델 모두 이 규칙을 따른다.
- 외부 API 응답 모델은 `*Response`로 끝낸다.
- 레이어 간 이동을 위한 내부 모델은 `*Dto`로 끝낸다. 내부 DTO는 FastAPI `Query`, `Depends` 같은 프레임워크 의존성을 갖지 않는다.
- 서비스 단건 조회 메서드는 `get_*`를 사용한다.
- 서비스 컬렉션 조회에서 필터, 페이지, 정렬 조건이 있으면 `search_*`를 사용한다.
- 특정 부모 자원이나 식별자 기준으로 컬렉션을 찾으면 `find_*_by_*`를 사용한다.
- `list_*`는 조건 없는 단순 나열일 때만 사용하고, 상태 필터나 페이지 조건이 있는 조회에는 사용하지 않는다.
- DB를 사용하는 서비스 메서드는 기본적으로 `db: Session`을 첫 번째 파라미터로 받는 `@staticmethod` 형태를 사용한다.
- 외부 클라이언트, 발송기, 또는 요청 동안 유지해야 할 상태를 보유하는 서비스만 인스턴스 형태를 허용한다.
- 입력 형식 검증은 schema, 비즈니스 규칙 검증과 엔티티 상태 변경은 domain model이 담당한다.
- 도메인 메서드는 동사와 자원을 포함해 이름만으로 수행 의도가 드러나게 작성한다.
- service는 조회, 권한 확인, 트랜잭션 조율을 담당하고 엔티티 필드를 직접 나열해 변경하지 않는다.
