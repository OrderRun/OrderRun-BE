# 사용자 서비스 책임 경계

## 문제 정의

기존 `UserAuthService`는 전화번호 인증, 사용자 인증, 프로필 변경, FCM 토큰, 회원 탈퇴를 모두 처리했다.
서로 다른 변경 이유와 트랜잭션 경계가 한 클래스에 결합되어 수정 범위와 회귀 위험이 커졌다.

## 결정

- `PhoneVerificationService`는 인증 레코드의 생성·검증·만료와 SMS 작업 등록을 담당한다.
- `UserAuthService`는 회원가입, 로그인, 토큰 발급·갱신 유스케이스를 조율한다.
- `UserProfileService`는 사용자 조회와 이름·알림·FCM 토큰 변경을 담당한다.
- `UserWithdrawalService`는 탈퇴 정책 검사, 활동 취소, 개인정보 정리를 하나의 트랜잭션으로 조율한다.
- 인증번호 생성, 해시, TTL 같은 순수 정책은 기존 `phone_verification` 모듈에 유지한다.

DB만 사용하는 서비스는 `db: Session`을 첫 인자로 받는 정적 메서드를 사용한다.
외부 발송기 상태를 보유하는 `PhoneVerificationService`만 인스턴스로 생성한다.

## 트랜잭션 경계

- 인증번호 발송은 레코드를 먼저 커밋한 뒤 SMS 백그라운드 작업을 등록한다.
- 코드 불일치와 만료는 실패 상태를 보존하기 위해 검증 서비스에서 커밋한다.
- 코드 검증 성공은 즉시 커밋하지 않고 회원가입 또는 로그인 최종 변경과 함께 커밋한다.
- 로그인 중 FCM 토큰 upsert는 `UserAuthService` 트랜잭션에 참여한다.
- 회원 탈퇴의 활동 취소, 부속 데이터 삭제, 사용자 익명화는 전부 성공할 때만 커밋한다.

## 고려한 대안

- 기존 서비스를 facade로 유지하면 호출부 전환은 줄지만 책임 경계가 계속 노출되므로 사용하지 않는다.
- 토큰 전용 서비스는 현재 정책이 작고 `core.security`로 이미 분리되어 있어 추가하지 않는다.
- FCM 전용 서비스는 다중 기기나 토큰 생명주기 정책이 생길 때 도입한다.

## 호환성과 리스크

외부 API, DTO, 오류 코드와 DB 스키마는 변경하지 않는다. 서비스 내부 호출 인터페이스는 변경된다.
인증 실패 상태의 독립 커밋은 보안 정책상 의도된 예외이며, 성공 경로는 상위 유스케이스와 원자성을 유지한다.
