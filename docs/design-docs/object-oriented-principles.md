# 객체지향 설계 원칙

## 목적

이 문서는 OrderRun-BE가 따르는 객체지향 설계 원칙과 Python 적용 기준을 기록한다.
새 기준은 실제 설계 결정과 함께 검증한 뒤 세부 원칙으로 추가한다.

## 대전제

- 상태와 동작을 책임에 맞게 묶고, 각 구성 요소의 책임을 명확하게 유지한다.
- 객체지향을 클래스 수를 늘리는 방식으로 해석하지 않는다.
- 기존 구조로 책임을 명확하게 표현할 수 있다면 불필요한 클래스와 추상화를 만들지 않는다.
- 동일한 의미를 표현할 수 있는 선택지 중 가장 단순하고 명시적인 형태를 우선한다.

## 메서드 선택

### 인스턴스 메서드

- 인스턴스가 보유한 상태를 읽거나 변경하는 동작에 사용한다.
- `self.db`, `self.sms_sender`처럼 생성 시 주입된 의존성이 필요한 동작은 인스턴스 메서드로 작성한다.
- 호출 대상 인스턴스가 `self`로 자동 전달된다는 점을 전제로 한다.

### 정적 메서드

- 인스턴스 상태를 사용하지 않지만 개념적으로 해당 클래스의 책임에 속하는 동작에 사용한다.
- 호출마다 계산하거나 새로운 값을 생성하는 동작은 고정 상수와 구분한다.
- 단순히 `self`를 사용하지 않는다는 이유만으로 정적 메서드로 만들지 않고, 클래스 책임과의 응집도를 함께 판단한다.
- 클래스 책임과 직접적인 관련이 없거나 여러 구성 요소에서 공유하는 동작은 모듈 함수 또는 별도 모듈을 고려한다.

인스턴스 상태가 필요 없는 동작이 여러 책임과 연결되면 정적 메서드를 한 클래스에 모으지 않는다.
인증번호 생성은 인증 정책 모듈, 토큰 만료시간 계산은 보안 모듈, 현재 시각 생성은 시간 모듈에 둔다.

## 상수 선택

- 변경하지 않을 정책값은 대문자 모듈 상수로 선언한다.
- 하나의 클래스에서만 사용하더라도 값 자체가 상태나 동작을 갖지 않으면 별도의 상수 보관 클래스를 만들지 않는다.
- 관련된 여러 구성 요소가 같은 정책값을 공유하게 되면 책임이 명확한 정책 모듈로 분리한다.
- 환경이나 요청에 따라 값이 달라지거나 행위와 교체 가능성이 필요할 때만 설정 객체나 정책 객체 도입을 고려한다.

인증번호 유효시간은 `app.services.phone_verification`에서 다음 기준을 따른다.

- `VERIFICATION_CODE_TTL_MINUTES`가 안내 문구에 사용하는 정책값을 제공한다.
- `VERIFICATION_CODE_TTL`이 만료 시각 계산에 사용하는 시간 값을 제공한다.
- 인증번호 생성, 해시, 안내 문구, 테스트 코드 판단 함수와 함께 하나의 인증 정책을 표현한다.

## 책임별 모듈 배치

- 비즈니스 규칙과 정책값은 관련 서비스 레이어에 둔다.
- 애플리케이션 전반에서 사용하는 시간, 보안, 설정 같은 기반 기능은 `core`의 책임별 모듈을 재사용한다.
- 서비스 클래스는 저장소와 외부 시스템을 사용하는 흐름 및 트랜잭션 조율에 집중한다.
- 서로 다른 책임의 함수를 한곳에 모으기 위한 포괄적인 `utils` 패키지는 만들지 않는다.

## 변경 시 확인

- 새 메서드가 인스턴스 상태나 주입된 의존성을 실제로 사용하는가?
- 값이 고정 정책인지, 호출 시 계산되는 동작인지 구분했는가?
- 클래스가 책임과 동작을 묶는 객체인지, 상수를 감싸기만 하는 이름 공간인지 확인했는가?
- 새 추상화가 기존 모듈이나 클래스보다 책임을 더 명확하게 만드는가?
- 이 문서의 기존 원칙과 다른 선택이 필요하면 예외의 근거를 관련 설계 문서에 기록했는가?
