# 핵심 신념

## 엔지니어링 신념

- 프레임워크 기교보다 도메인 모델이 더 중요하다.
- 넓은 공용 추상화보다 작고 명시적인 모듈이 오래 간다.
- API 형태는 데이터베이스 편의보다 사용자 흐름을 따라야 한다.
- 상태 전이는 별도의 검증과 테스트를 받을 가치가 있다.
- 생성 문서는 코드와 스펙에 계속 맞아 있을 때만 의미가 있다.

## 문서화 신념

- 제품 의도, 기술 설계, 실행 상태는 서로 다른 문서에 둔다.
- 한 문서는 하나의 핵심 질문에 답하도록 만든다.
- 길지만 방치된 문서보다 짧아도 유지되는 문서가 낫다.
- 파생 문서는 임의 수정보다 재생성 또는 재정렬을 우선한다.

## 전달 신념

- 기능 확장보다 핵심 루프의 신뢰성을 먼저 확보한다.
- 새로운 헬퍼를 덧대기보다 삭제와 단순화를 우선한다.
- 열린 리스크는 다음 작업이 바로 찾을 수 있는 위치에 남긴다.
